# Reflection

## 1. Hardest bug this week

The hardest issue was the lead capture flow failing after the UI looked complete. The form could show a success-like state, but Supabase returned that `public.leads` could not be found in the schema cache. My first hypothesis was that the API payload was wrong, so I checked what the route expected: `audit_id`, `email`, optional company/role/team size fields, and email delivery status. The next hypothesis was row-level security, because an insert can fail even when a table exists if the policy is missing. The actual root cause was more basic: the table did not exist in Supabase yet. I used AI to inspect the route and generate the expected SQL, but I did not just trust the output. One AI-generated SQL attempt used `timestz`, which PostgreSQL rejected because the correct type is `timestamptz`. I corrected that, added primary key, timestamps, indexes, RLS, insert policy, and `notify pgrst, 'reload schema';`. The fix was production-real rather than clever: match the database schema to the code, keep public insert permissions minimal, and handle email failure separately from lead storage.

## 2. Decision I reversed mid-week

The main decision I reversed was how much responsibility to give the LLM. At first, because the product is about AI spend and the assignment requires an AI-generated summary, it was tempting to let an LLM produce most of the recommendations. I reversed that because the product makes financial claims. If ModelMeter says a team can save `$960/year`, the user should be able to understand exactly where that number came from: seat count, plan benchmark, overlap, or a documented assumption. A free-form LLM answer might sound smart but would be harder to audit and easier to overclaim. I kept the core audit engine deterministic and used AI in two separate ways: first, as a development assistant to help build the repository; second, at runtime only for a short summary paragraph based on precomputed data. I also reversed the PDF approach. The first export direction felt too much like debug output or a screenshot substitute. I moved it toward a structured server-generated report with sections, tables, recommendations, timestamp, and report link. Both reversals came from the same product judgment: for this assignment, trust matters more than appearing magical.

## 3. What I would build in week 2

Week 2 would start with real user validation, because that is the weakest remaining part of the submission. The `USER_INTERVIEWS.md` file is intentionally not filled with fake quotes; I would talk to three actual founders, engineering managers, or finance/ops people who have paid for Cursor, ChatGPT, Claude, Copilot, or API usage. I would use those conversations to find contradictions in the product, not just confirmation. After that, I would add CSV import for spend rows from Ramp, Brex, or a spreadsheet, because manual entry is acceptable for an MVP but not ideal for repeated finance use. I would also build a small internal Credex lead view showing report URL, modeled savings, tools involved, high-savings status, and email delivery status. For the audit engine, I would expand tests around API-heavy teams, annual contracts, coding-heavy teams, and small teams that are actually already optimized. I would avoid turning it into a full procurement platform. The next version should still be a fast acquisition tool: finish an audit, understand the savings, share the report, and route high-savings teams toward a Credex conversation.

## 4. How I used AI tools

I used AI tools heavily and intentionally because I had exams and limited candidate-directed build time. The repository was built primarily with Codex/ChatGPT-style assistance through iterative prompts, review, and correction rather than a single one-shot generation. AI helped generate and edit the Next.js app, React components, deterministic audit engine, Supabase schema, lead capture flow, Resend email integration, PDF export, tests, CI workflow, and documentation drafts. My role was to provide the assignment context, decide the product direction, review the output, give real error messages, and keep the project aligned with the brief. The most important thing I did not trust AI with was the product's financial truth. The audit math is deterministic TypeScript, not runtime LLM reasoning. Pricing assumptions are documented in `PRICING_DATA.md`, and the optional Anthropic call only summarizes precomputed findings. One specific time AI was wrong was the Supabase SQL typo: it generated `timestz`, which failed because PostgreSQL expects `timestamptz`. I caught that through the actual SQL error and corrected the schema. I also pushed back on vague recommendation language because it sounded confident without being traceable. So the honest version is: AI did most of the implementation labor, while my contribution was direction, review, constraint-setting, and correction.

## 5. Self-rating

**Discipline: 6/10.** I stayed focused on the required product and did not expand it into unnecessary features, but I did not meet the spirit of a full seven-day manual work process because exams limited my time and AI handled most implementation.  
**Code quality: 7/10.** The app now builds, lints, tests, and keeps the audit logic deterministic, but because much of the code was AI-generated I would want more manual review and integration tests before treating it as production software.  
**Design sense: 8/10.** The final product feels like a restrained utility instead of an overhyped AI landing page, and the audit-first flow is the right interaction for this use case.  
**Problem-solving: 7/10.** I used real failures, especially Supabase schema and email-flow issues, to guide fixes instead of assuming generated code was correct.  
**Entrepreneurial thinking: 7/10.** The product has a believable Credex acquisition wedge: find teams with high modeled AI savings and route them toward discounted credits. I would rate this higher only after replacing the interview template with three real conversations and using those findings to change the product.
